# トップページ ヒーロー背景動画 構想案・実装計画

作成日: 2026-07-17
ステータス: 設計検討のみ（実装は次フェーズ）
対象ブランチ: `docs/home-hero-video-plan`

本ドキュメントは、Ban.Tai公式HPのトップページに背景動画を導入するための構想と実装計画をまとめたものです。今回のPRでは本ドキュメントの作成のみを行い、HTML/CSS/JSの変更、動画・画像素材の追加は行いません。

---

## 1. 導入目的

Ban.Taiは教育現場40年の経験を土台に、先生の仕事を支える製品を提供しています。トップページに黒・金・白を基調とした静かな背景動画を加えることで、次の3点を伝えやすくします。

教育現場を長く支えてきた実績と信頼感。派手さではなく上質さで魅せるブランドの質感。五線紙・週案表・カレンダー・座席表といった、先生が日々使う道具への敬意。

動画はあくまで背景であり、主役はテキストと製品情報です。動画によって文字の可読性やページ速度が損なわれることは避けます。

---

## 2. 推奨デザイン案

黒または濃紺（例: `#0a0a0f` 〜 `#12121a` 程度の非常に暗いトーン）を基調とし、その上を金色の細い線・光の粒子がゆっくり移動する構成を推奨します。動きの参考イメージは以下の通りです。

五線紙の罫線が横方向にごくわずかに流れる、あるいは週案表や座席表のグリッド線が淡く浮かび上がっては消える。金色の光の筋が1本、画面を斜めにゆっくり横切る（1ループ8〜15秒）。粒子やパーティクルを使う場合も、密度を低く保ち「静電気のような輝き」程度に留める。

意図的に避ける表現として、人物・子どもの顔・学校名や名簿等の個人情報が映るもの、カメラのパン/ズーム/切り替えを伴う実写的な演出、明滅・点滅・高速な動きを伴うもの、をすべて除外します。あくまで「抽象化された教育ツールの気配」を表現し、実在の教室や人物を想起させる具体的な描写は行いません。

素材の出どころについては、自作のモーショングラフィックス（After Effects / Lottie / CSS・Canvasアニメーション等での自製）を基本方針とし、外部素材（ストック動画・AI生成動画・他者の著作物）を無断で使用することはしません。素材を外部から調達する場合は、ライセンス条件を確認し、本ドキュメントとは別に素材調達の承認を得たうえで着手します。

---

## 3. 実装しない方がよいパターン

以下のパターンは、今回のブランド方針・パフォーマンス方針に反するため実装しません。

トップページ全体（フッターや製品一覧セクションまで）に動画を敷く案は、読み込み負荷が大きくテキスト可読性を損なうリスクが高いため見送ります。ヒーローセクション以外への拡大は行いません。

自動再生の実写ドローン映像や学校紹介VTR的な素材は、教室・人物が映り込むリスクが高く、個人情報保護の観点からも避けます。

音声付き動画、8秒未満の慌ただしいループ、派手なパーティクル爆発やグリッチ演出は、静かで信頼感のあるブランドイメージと相容れないため実装しません。

10MBを超えるような高解像度・高ビットレートの動画は、GitHub Pages配信の性質上（後述）採用しません。

スマートフォンでの動画自動再生を無条件に強制する実装（データ通信量への配慮なし）も避け、後述のスマホ対応方針に従います。

---

## 4. 技術仕様案

対象範囲はトップページのヒーローセクションのみとし、それ以外のセクションやページには影響を与えません。

動画の長さは8〜15秒程度のシームレスループとし、音声トラックは含めません（無音、`muted`前提）。

video要素の属性は `autoplay muted loop playsinline` を基本とし、`controls` は付与しません。ユーザー操作による一時停止UIを設けるかどうかは次PR以降の検討事項とします。

ファイル容量の目安は3〜6MB以内を目標とし、上限を10MB程度とします。解像度は1920x1080を上限とし、必要であれば1280x720程度にダウンスケールしてファイルサイズを抑えます。

コーデックは `webm`（VP9、圧縮効率が高い）を第一候補、`mp4`（H.264）をSafari等の互換性確保のためのフォールバックとして併用する案を推奨します。`<video>` 要素内に `<source>` を複数指定し、ブラウザが対応形式を自動選択する形とします。

```
<source src="/assets/video/hero-bg.webm" type="video/webm">
<source src="/assets/video/hero-bg.mp4" type="video/mp4">
```

## 5. HTML構造案（実装時のイメージ、今回は反映しません）

```html
<section class="hero" aria-label="Ban.Tai について">
  <div class="hero-media" aria-hidden="true">
    <video
      class="hero-video"
      poster="/assets/images/hero-poster.jpg"
      autoplay
      muted
      loop
      playsinline
      preload="none">
      <source src="/assets/video/hero-bg.webm" type="video/webm">
      <source src="/assets/video/hero-bg.mp4" type="video/mp4">
    </video>
    <div class="hero-overlay"></div>
  </div>

  <div class="hero-content">
    <h1>...</h1>
    <p>...</p>
  </div>
</section>
```

動画は `aria-hidden="true"` の装飾コンテナ内に配置し、スクリーンリーダーには読み上げさせません。実際のメッセージ・見出し・CTAは `hero-content` 側の通常のテキストとして実装し、動画の有無に関わらず内容が伝わる構造を維持します。

## 6. CSS設計案（実装時のイメージ、今回は反映しません）

```css
.hero {
  position: relative;
  overflow: hidden;
}

.hero-media {
  position: absolute;
  inset: 0;
  z-index: 0;
}

.hero-video {
  width: 100%;
  height: 100%;
  object-fit: cover;
}

.hero-overlay {
  position: absolute;
  inset: 0;
  background: rgba(0, 0, 0, 0.55); /* 文字の可読性確保のための黒半透明オーバーレイ */
}

.hero-content {
  position: relative;
  z-index: 1;
}
```

オーバーレイの不透明度（案では0.5〜0.6程度）は、実際の金色ラインの輝度やテキストのコントラスト比（WCAG AA基準、4.5:1以上を目安）を見ながら調整します。

## 7. アクセシビリティ対応

`prefers-reduced-motion: reduce` が指定された環境では、動画を停止し静止画（poster画像、または同一デザインの静止フレーム）に切り替えます。実装イメージは以下の通りです。

```css
@media (prefers-reduced-motion: reduce) {
  .hero-video {
    display: none;
  }
  .hero-media {
    background-image: url("/assets/images/hero-poster.jpg");
    background-size: cover;
    background-position: center;
  }
}
```

または、JS側で `matchMedia('(prefers-reduced-motion: reduce)')` を判定し、`video.pause()` を呼ぶ実装でも代替可能です。CSSのみで完結する前者の方が保守性が高いため推奨します。

動画コンテナには `aria-hidden="true"` を付与し、装飾目的であることを明示します。動画が読み込まれない・再生されない場合でも、見出しやCTAボタンの操作性・可読性が損なわれないことを実装後に確認します。

---

## 8. パフォーマンス対策

poster画像は必須とします。動画の読み込み前・再生開始前・`prefers-reduced-motion`環境・読み込み失敗時のフォールバックとして機能するため、動画本体とは別に軽量な静止画（JPEG/WebP、200KB程度以内を目標）を用意します。

`preload="none"` または `preload="metadata"` を基本とし、ページの初期表示（LCP）を動画が阻害しないようにします。動画自体はヒーローセクションの背景であり、LCP対象要素（見出しテキストなど）より優先度を下げます。

`loading` の遅延読み込みに相当する制御として、Intersection Observer等でヒーローセクションが表示された際にのみ動画をロード・再生する案も次PR以降で検討します（初回実装ではシンプルに`preload="none"`のみとする案も可）。

動画ファイルは事前に十分に圧縮し、上記3〜6MBの目安を超えないことをコミット前に確認します。

---

## 9. スマホ対応

スマートフォンでは、データ通信量への配慮と自動再生制限（iOS Safari等では`playsinline`+`muted`が無いと自動再生されない）を踏まえ、次の2案を比較検討します。

案A: スマホでも同一動画を再生する。`playsinline muted autoplay loop`を維持し、容量を抑えた低解像度版（720p程度）を別途用意して出し分ける。

案B: スマホでは動画を読み込まず、poster静止画のみを表示する。CSSの`@media (max-width: 768px)`等でvideo要素を`display:none`にし、`hero-media`の背景画像をposterに切り替える。

通信量とバッテリー消費を優先するなら案B、ブランド体験の一貫性を優先するなら低容量版を用意した案Aが候補になります。どちらを採用するかは、次PRの着手前に判断する事項とします（本ドキュメントでは両案を提示するに留めます）。

## 11. mp4 / webm の扱いとGitHub Pages配信の注意点

GitHub Pagesは静的ファイルホスティングであり、動画配信に特化したCDNではありません。大容量ファイルはリポジトリサイズの肥大化とページ表示速度の低下を招くため、以下に注意します。

GitHub単体のリポジトリ上限（1ファイルあたり100MB、リポジトリ全体の推奨上限1GB程度）を踏まえ、動画は数MB単位に厳しく圧縮します。Git LFSの利用も選択肢ですが、GitHub Pagesの無料枠ではLFS経由のファイルがそのまま配信されない場合があるため、LFSを使う場合は配信可否を事前検証します。

`webm`と`mp4`の二重管理は容量が単純に倍になる点に留意し、圧縮設定（ビットレート、フレームレート、解像度）を両形式で揃えて管理します。フレームレートは24fps程度に抑えても静かな動きの表現には十分です。

GitHub Pagesにはキャッシュ制御ヘッダーのカスタマイズがしづらい制約があるため、ファイル名にハッシュを付与する、または更新頻度が低いことを前提にブラウザキャッシュに任せる方針とします。

## 12. まず静止画像で試す案 と 動画を入れる案の比較

**静止画像案**: 金色の線や五線紙モチーフを描いた1枚の高品質な静止画（またはCSSグラデーション+SVGパターン）を背景にする。実装コストが低く、容量もごく小さく（数十〜数百KB）、パフォーマンスへの影響も最小。将来的に動画へ差し替える際の土台（poster画像）としてもそのまま使える。静けさ・上質感の表現は動画よりやや弱くなるが、CSSアニメーション（例: 背景グラデーションのゆっくりした移動、SVGラインの`stroke-dashoffset`アニメーション）を組み合わせれば、動画に近い「静かな動き」を軽量に再現できる。

**動画案**: より豊かな質感・奥行きを表現できるが、素材制作コスト（撮影・モーショングラフィックス制作）、容量管理、`prefers-reduced-motion`対応、スマホ対応など考慮点が多く、実装・運用コストが高い。

推奨は、まず静止画像＋軽量CSSアニメーション案で実装し、ブランド表現として十分かを確認したうえで、必要であれば動画に置き換える段階的アプローチです。静止画案は動画案のposter画像としてもそのまま流用できるため、投資が無駄になりません。

## 13. 段階的な実装手順（次フェーズ以降の想定）

まず本設計書のレビュー・承認を行います（本PR）。

次に、静止画像＋CSSアニメーション版をヒーローセクションに実装し、可読性・パフォーマンス（Lighthouse等での計測）を確認します（次PR案）。

静止画版の評価が良ければ、動画素材（自作モーショングラフィックス）を制作し、`prefers-reduced-motion`対応・スマホ出し分け・poster設定を含めて動画版を実装します（さらに次のPR）。

最後に、実機（iOS Safari、Android Chrome）での自動再生・容量・バッテリー影響を検証し、必要に応じてスマホでは静止画に切り替える最終判断を行います。

## 14. 次PRで実装する場合の作業範囲（最小PR）

次PRを「静止画像＋CSSアニメーション版」とする場合の最小範囲は次の通りです。

変更対象は `index.html` のヒーローセクション（`hero-media`コンテナの追加）、`assets/style.css`（`.hero-media`・`.hero-overlay`・`prefers-reduced-motion`対応の追加）のみとし、他ページ・他セクションには触れません。

追加ファイルは背景用の静止画像（またはSVG）1点と、そのライセンス・出所を明記した短いメモ（`docs/design/`配下）のみとします。動画ファイル・外部素材はこの段階でも追加しません。

このPRとは別に、動画素材を導入する場合は改めて素材の出所・ライセンス確認を行うPR（または本ドキュメントの改訂）を先に立て、承認後に着手します。
