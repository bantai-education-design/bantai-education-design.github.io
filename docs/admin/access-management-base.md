# Ban.Tai 公式HP アクセス管理ベース設計・運用メモ

## 1. 管理ベースの目的

Ban.Tai 公式HPのアクセス状況、商品ページの閲覧状況、外部販売ページへの導線、教科書会社リンク集の利用状況を、管理者が週次・月次で確認できるようにするための設計メモです。

この管理ベースでは、公開サイト上に管理画面を作成しません。管理者は、GitHub Insights、Google Analytics 4、Looker Studio、Googleスプレッドシートなどの外部管理ツールを使って状況を確認します。

今回の範囲では、ログイン機能、個人情報、購入者情報、ライセンスキーは扱いません。GA4測定ID、APIキー、サービスアカウントキーなどの秘密情報もリポジトリに保存しません。

## 2. 確認するアクセス指標

管理者が定期的に確認する主な指標は次のとおりです。

- トップページ閲覧数
- 商品一覧ページ閲覧数
- 各商品詳細ページ閲覧数
- 教科書会社リンク集ページ閲覧数
- BOOTHボタンクリック数
- Vectorボタンクリック数
- noteボタンクリック数
- 無料ダウンロードボタンクリック数
- 教科書会社公式HPボタンクリック数
- 年間指導計画ページボタンクリック数
- 参照元
- 人気ページ
- 検索キーワード利用状況
- 商品フィルター利用状況

アクセス数だけでなく、「どのページを見た人が、どの外部導線をクリックしたか」を把握できる状態を目指します。

## 3. GitHub Insightsで確認できる項目

GitHub Pagesで公開している場合、GitHub Insightsでは主にリポジトリ単位の状況を確認します。公式HPの細かいページ別アクセス解析には向かないため、GA4やLooker Studioと役割を分けます。

GitHub Insightsで見るべき項目は次のとおりです。

- リポジトリのアクセス傾向
- クローン数、ユニーククローン数
- リポジトリ閲覧数、ユニーク閲覧者数
- 参照元サイト
- 人気コンテンツ
- GitHub Pages関連の公開ブランチ、公開設定
- デプロイや更新頻度の確認

GitHub Insightsは、公式HPそのもののアクセス解析というより、リポジトリや公開運用の健全性確認に使います。

## 4. Google Analytics 4で確認したい項目

GA4では、ページ閲覧、外部リンククリック、商品検索、商品フィルターの利用状況を確認します。

確認したい主な項目は次のとおりです。

- ページ別閲覧数
- ユーザー数
- 新規ユーザー数
- 参照元、メディア、キャンペーン
- デバイス種別
- 地域の大まかな傾向
- 人気ページ
- 離脱が多いページ
- 外部販売ページへのクリック数
- 商品検索の利用回数
- 商品フィルターの利用回数

GA4イベント名案は次のとおりです。

| イベント名 | 用途 |
| --- | --- |
| `click_booth` | BOOTHボタンのクリック |
| `click_vector` | Vectorボタンのクリック |
| `click_note` | noteボタンのクリック |
| `click_free_download` | 無料ダウンロードボタンのクリック |
| `click_textbook_publisher` | 教科書会社公式HPボタンのクリック |
| `click_textbook_plan` | 年間指導計画ページボタンのクリック |
| `product_search` | 商品検索の実行 |
| `product_filter` | 商品フィルターの利用 |

イベントには、個人を特定できる情報を含めません。商品名、商品カテゴリ、リンク種別、ページ種別など、管理に必要な最小限の情報だけを使います。

### 現状確認

2026年7月12日時点の公式HPリポジトリ確認では、`tools/tokyo-school-address/index.html` に `G-REPLACE_ME` のGA4プレースホルダーと `gtag` 記述があり、複数ページで `/assets/js/analytics-events.js` が読み込まれています。一方で、本番用のGA4測定IDは確認できませんでした。

次フェーズでは、現在のプレースホルダーと共通イベント送信スクリプトの役割を確認し、GA4測定IDをリポジトリに直書きしない方法で公式HP全体の計測方針を整理します。

## 5. Looker StudioまたはGoogleスプレッドシートで管理する項目

Looker Studioは、GA4のデータを見やすいダッシュボードにまとめる用途で使います。Googleスプレッドシートは、週次確認、月次振り返り、改善メモ、リリース履歴を管理する用途で使います。

Looker Studioで見る項目案は次のとおりです。

- 週次アクセス推移
- ページ別閲覧数ランキング
- 商品別ページ閲覧数
- 外部導線クリック数
- 参照元別アクセス
- 教科書会社リンク集のクリック状況
- 検索、フィルター利用状況

Googleスプレッドシートでは、数値だけでなく「なぜ変化したか」「次に何を直すか」を残します。

## 6. 商品別に確認する指標

商品ごとに、閲覧から外部リンククリックまでの流れを確認します。

| 指標 | 確認内容 |
| --- | --- |
| 商品詳細ページ閲覧数 | どの商品が見られているか |
| BOOTHクリック数 | BOOTH販売ページへの移動が発生しているか |
| Vectorクリック数 | Vector販売ページへの移動が発生しているか |
| noteクリック数 | note記事や補足情報への移動が発生しているか |
| 無料ダウンロードクリック数 | 無料版、体験版、資料への関心があるか |
| 検索キーワード | どの商品名や用途で探されているか |
| フィルター利用 | 学年、教科、用途などの絞り込みが使われているか |

商品別の確認では、販売数や購入者情報は扱いません。公式HP上で確認できる導線の利用状況だけを対象にします。

## 7. BOOTH / Vector / note クリック導線の確認項目

外部販売・外部発信ページへの導線は、次の観点で確認します。

- どのページからクリックされたか
- どの商品に関連するクリックか
- どの外部サービスへのクリックか
- 週次でクリック数が増減しているか
- ページ閲覧数に対してクリック率が極端に低くないか
- リンク切れやリンク先変更が発生していないか

管理上の区分案は次のとおりです。

| 導線 | GA4イベント名案 | 補足 |
| --- | --- | --- |
| BOOTH | `click_booth` | 販売ページへの導線 |
| Vector | `click_vector` | 販売ページ、配布ページへの導線 |
| note | `click_note` | 告知、説明、補足記事への導線 |
| 無料ダウンロード | `click_free_download` | 無料版、体験版、資料への導線 |

## 8. 教科書会社リンク集の確認項目

教科書会社リンク集では、公式HPリンクと年間指導計画ページリンクの利用状況を確認します。

確認項目は次のとおりです。

- 教科書会社リンク集ページ閲覧数
- 教科書会社公式HPボタンクリック数
- 年間指導計画ページボタンクリック数
- 会社別クリック数
- 教科、学年、用途別のクリック傾向
- リンク切れ、URL変更の有無

GA4イベント名案は次のとおりです。

- `click_textbook_publisher`
- `click_textbook_plan`

イベントパラメータ案は次のとおりです。

| パラメータ | 内容 |
| --- | --- |
| `publisher_name` | 教科書会社名 |
| `link_type` | `official_site` または `annual_plan` |
| `subject` | 教科 |
| `page_location` | クリック元ページ |

個人情報や学校名など、利用者を特定できる情報は記録しません。

## 9. 週1回の確認手順

週1回、次の順番で確認します。

1. GitHub Pagesが公開状態を維持しているか確認する。
2. GitHub Insightsでリポジトリ閲覧数、参照元、人気コンテンツを確認する。
3. GA4でトップページ、商品一覧、商品詳細、教科書会社リンク集の閲覧数を確認する。
4. GA4でBOOTH、Vector、note、無料ダウンロードのクリック数を確認する。
5. GA4で教科書会社公式HP、年間指導計画ページのクリック数を確認する。
6. 検索キーワード、商品フィルター利用状況を確認する。
7. Googleスプレッドシートの `01_週次サマリー` に数値と気づきを記録する。
8. リンク切れや導線の弱いページがあれば `06_改善メモ` に記録する。

週次確認では、細かい分析よりも「異常がないか」「改善候補があるか」を見ることを優先します。

## 10. 月1回の振り返り手順

月1回、次の順番で振り返ります。

1. 直近4週分のアクセス数、外部導線クリック数を比較する。
2. 人気ページと低迷ページを確認する。
3. 商品別に閲覧数と外部リンククリック数を確認する。
4. 教科書会社リンク集の利用状況を確認する。
5. 検索キーワード、商品フィルター利用状況から、探されている商品や用途を確認する。
6. リリース履歴や告知タイミングとアクセス変化を照合する。
7. 次月に改善するページ、導線、文言、リンクを決める。
8. `06_改善メモ` にCodexへ依頼する作業候補を整理する。

月次振り返りでは、数値の増減だけでなく、更新内容や外部告知との関係を見ます。

## 11. Codexに依頼する改善作業の書き方

Codexへ改善作業を依頼するときは、対象、目的、変更してよい範囲、触ってはいけない範囲を明確にします。

依頼例:

```text
Ban.Tai公式HPの商品詳細ページについて、GA4の週次確認で閲覧数はあるがBOOTHクリックが少ない商品があります。

対象:
- 商品Aの詳細ページ
- 商品Bの詳細ページ

目的:
- 外部販売ページへの導線を分かりやすくする

やってほしいこと:
- 現在のHTML/React構成を確認
- 商品詳細ページ内の販売ボタンの位置、文言、リンク状態を確認
- 公開サイトの見た目を大きく変えずに改善案を出す

注意:
- GA4測定IDやAPIキーはリポジトリに書かない
- 個人情報、購入者情報、ライセンスキーは扱わない
- 管理画面は作らない
```

GA4イベント実装を依頼する場合は、測定IDを直接書かず、環境変数やGitHub Pages側の安全な設定方法を検討対象にします。

## 12. 注意事項

- GitHub Pages上に公開の管理画面を作成しない。
- ログイン機能や個人情報を扱う管理画面は作成しない。
- GA4測定ID、APIキー、サービスアカウントキーなどの秘密情報をリポジトリに書かない。
- 個人情報、購入者情報、ライセンスキーを扱わない。
- Googleスプレッドシートに個人情報や購入者情報を記録しない。
- GA4イベントには個人を特定できる値を入れない。
- 公開サイトの見た目を変更する作業と、管理ベースの設計作業を分ける。
- GA4タグが未実装の場合は、次フェーズで実装する。
- GA4タグが既にある場合は、測定IDをリポジトリ内に直書きしていないか確認する。

## Googleスプレッドシート管理ベース案

Googleスプレッドシートは、GA4やGitHub Insightsの数値を手動または半自動で転記し、週次・月次の判断を残すために使います。

### `01_週次サマリー`

| 列 | 内容 |
| --- | --- |
| 週 | 対象週 |
| トップページ閲覧数 | GA4から確認 |
| 商品一覧ページ閲覧数 | GA4から確認 |
| 商品詳細ページ合計閲覧数 | GA4から確認 |
| 教科書会社リンク集閲覧数 | GA4から確認 |
| 外部導線クリック合計 | BOOTH、Vector、note、無料ダウンロードの合計 |
| 主な参照元 | GA4またはGitHub Insightsから確認 |
| 気づき | 増減理由、告知、更新内容など |
| 次の対応 | 改善候補 |

### `02_ページ別アクセス`

| 列 | 内容 |
| --- | --- |
| 週 | 対象週 |
| ページ名 | トップ、商品一覧、商品詳細など |
| URL | 対象URL |
| 閲覧数 | GA4から確認 |
| ユーザー数 | GA4から確認 |
| 前週比 | 増減確認 |
| メモ | 変化理由、改善候補 |

### `03_外部導線`

| 列 | 内容 |
| --- | --- |
| 週 | 対象週 |
| 導線種別 | BOOTH、Vector、note、無料ダウンロード |
| イベント名 | GA4イベント名 |
| クリック元ページ | クリックされたページ |
| 商品名 | 関連商品 |
| クリック数 | GA4から確認 |
| 前週比 | 増減確認 |
| リンク確認 | 正常、要確認、修正済み |

### `04_商品別チェック`

| 列 | 内容 |
| --- | --- |
| 週 | 対象週 |
| 商品名 | 対象商品 |
| 商品カテゴリ | 教科、用途、シリーズなど |
| 商品詳細ページ閲覧数 | GA4から確認 |
| BOOTHクリック数 | GA4から確認 |
| Vectorクリック数 | GA4から確認 |
| noteクリック数 | GA4から確認 |
| 無料ダウンロードクリック数 | GA4から確認 |
| 検索キーワード | GA4イベントまたはサイト内検索ログから確認 |
| フィルター利用 | GA4イベントから確認 |
| 改善メモ | 導線、説明文、リンク状態など |

### `05_GitHub_Pages`

| 列 | 内容 |
| --- | --- |
| 確認日 | 確認した日 |
| 公開状態 | 正常、要確認 |
| 公開ブランチ | GitHub Pages設定 |
| 最新更新日 | リポジトリの更新日 |
| GitHub Insights閲覧数 | Insightsから確認 |
| 主な参照元 | Insightsから確認 |
| 人気コンテンツ | Insightsから確認 |
| メモ | デプロイ、リンク、公開設定の注意 |

### `06_改善メモ`

| 列 | 内容 |
| --- | --- |
| 記入日 | メモ作成日 |
| 優先度 | 高、中、低 |
| 対象ページ | 改善対象 |
| 課題 | 数値や確認結果から見えた課題 |
| 改善案 | 実施したい内容 |
| Codex依頼文 | そのまま依頼に使える文章 |
| 状態 | 未着手、対応中、完了、保留 |
| 完了日 | 対応完了日 |

### `07_リリース履歴`

| 列 | 内容 |
| --- | --- |
| リリース日 | 公開、更新した日 |
| バージョン | 任意の管理番号 |
| 変更内容 | 追加、修正、削除した内容 |
| 対象ページ | 変更したページ |
| 告知有無 | note、SNS、BOOTHなど |
| アクセス変化 | リリース後の変化 |
| メモ | 次回確認事項 |

## 次フェーズで実装すべきこと

次フェーズでは、次の作業を検討します。

1. GA4プロパティを作成する。
2. GA4測定IDをリポジトリに直書きしない実装方法を決める。
3. ページ閲覧計測の実装方針を決める。
4. 外部リンククリックイベントを実装する。
5. 商品検索、商品フィルターイベントを実装する。
6. Looker Studioのダッシュボード構成を作る。
7. Googleスプレッドシートの管理ベースを作る。
8. 週次・月次の運用テンプレートを実際の数値に合わせて調整する。
